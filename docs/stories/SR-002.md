## User Story
En tant qu'utilisateur, je veux que mon fichier `.sav` soit entièrement parsé afin que toutes les entités, liens drones, rails et zones soient disponibles via l'API.

## Critères d'acceptation
- [ ] Chaque entité du JSON est extraite et classifiée (basecore, machine, energy, infra, antenna, danger, loot)
- [ ] Les coordonnées X, Y, Z sont extraites pour chaque entité
- [ ] Les recettes actives sont extraites via regex sur `CrCraftingFragment`
- [ ] Le niveau d'infection est extrait via regex sur `CrBuildingInfection`
- [ ] Les liens drones (source → destination + item) sont extraits et stockés dans `drone_links`
- [ ] Les points de spline (DroneRail, Walkway) sont extraits et stockés dans `rail_splines` (JSONB)
- [ ] Les bounding boxes BaseCore sont extraites et stockées dans `base_zones`
- [ ] GET `/api/saves/{id}/entities` retourne toutes les entités (filtrable par `?cat=`)
- [ ] GET `/api/saves/{id}/links` retourne les flux drones
- [ ] GET `/api/saves/{id}/splines` retourne les rails
- [ ] GET `/api/saves/{id}/zones` retourne les zones
- [ ] GET `/api/saves/{id}/summary` retourne les statistiques agrégées
- [ ] Migrations Flyway V1 et V2 (schema + indexes)
- [ ] Tests unitaires SaveParserService (regex, classification, extraction)

## Notes techniques
- Parsing dans `SaveParserService.parseSavFile()`
- Patterns regex définis dans CLAUDE.md backend
- Classification par `entityConfigDataPath`
- Persister via repositories JPA en batch
